# 00｜新版总览：完全抛弃经验分段，把 E5.5 变成 xWPCache 的“因果监督”

这份 v2 方案替换旧版 `xwp_cache_detailed_plan.zip`。旧版里有一些“早期强制 refresh、中后期大胆 skip、尾部轻保护”的经验味道。现在根据 E5 和 E5.5 的真实结果，我们把路线改成更严谨的版本：

> **不再让 E5.5 直接告诉算法哪个 call 段该刷新。**  
> **E5.5 只作为连续 skip 的真实因果标签，用来验证和校准 xWPCache 的数学风险分数。**

也就是说，新版 E6 不应该写成：

```text
call 1-7 refresh
call 50-77 skip
call 88-97 max_age=2
```

而应该写成：

```text
每个 call 计算一个由 x-prediction 推导出的 cache risk。
连续 skip 时把 risk 按 ODE 误差形式累积。
当累计风险超过阈值时 refresh。
E5.5 负责检验这个累计风险是否真的预测连续 skip 后的最终感知损伤。
```

---

## 1. 为什么必须修改旧方案？

E5 告诉我们：单次 skip 的危险主要集中在早期，call 16 以后单点 skip 几乎无损。

但 E5.5 告诉我们：

> **单次 skip 安全，不代表连续 skip 安全。**

你上传的 E5.5 结果里，有一个非常重要的现象：

```text
某些 call 单独 skip 几乎没影响，
但是从同一区域连续 skip 4 / 8 / 12 个 call 后，最终图会出现明显 LPIPS / DINO 损伤。
```

这说明真实 cache 风险不是单步风险，而是**连续复用误差的累积**。

所以新版 xWPCache 必须从“单点判断”升级为：

```text
anchor-relative accumulated solver error
基于上一次 refresh anchor 的累计求解器误差
```

---

## 2. 新版主线一句话

> **xWPCache-v2：在 x-prediction 模型中，cache 误差来自复用过期 clean prediction。我们用 Wiener clean proxy 近似当前 clean prediction，并在感知频域空间中累积由 velocity conversion 推导出的 solver error；E5.5 的连续 skip PIS 用来验证这个累计风险是否对应真实最终感知损伤。**

这句话里有几个关键词：

| 关键词 | 含义 |
|---|---|
| x-prediction | 模型直接预测 clean image，即 `xhat` |
| Wiener clean proxy | 不完整 forward，用当前 `x_t` 通过频域滤波估计 clean image |
| perceptual frequency space | 用类似 DINO/LPIPS 的频率权重关注语义和纹理 |
| velocity conversion | PixelGen 把 `xhat` 转成 velocity，所以 clean prediction 误差会被 `1/(1-t)` 放大 |
| anchor-relative | 真实 cache 是一直复用上一次 refresh 的输出，所以要和 anchor 比，而不是只和上一 call 比 |
| accumulated solver error | 连续 skip 时误差会累积，不能只看单步距离 |
| E5.5 PIS | 连续 skip window 后最终图的真实感知损伤，是方法的因果标签 |

## 3. 新版实验路线

### Step A：整理已有 E5.5 标签

输入：

```text
E5.5 每个 window 的结果：
window_id, sample_id, start_call, end_call, window_len,
PIS_LPIPS, PIS_DINO, PSNR, SSIM
```

输出：

```text
Y(W) = 连续 skip window W 后的真实感知损伤
```

这一步不是新跑模型，只是把结果整理成标准标签表。

---

### Step B：重新 dump full trajectory

为了计算 xWPCache 的离线风险分数，需要每个 sample 的 full trajectory：

```text
call_index
timestep t
step size h
call_kind
x_t
xhat_full
```

如果当前 E5 / E5.5 没有完整保存这些东西，就需要补一个 trajectory dump 脚本。

---

### Step C：做 E6-D0 离线诊断

这是最重要的一步。先不要做 online cache。

对 E5.5 的每个连续 skip window W，计算多个候选数学风险分数：

```text
Raw-x_t risk
SEA-like risk
xWiener risk
xWiener + perceptual weight
xWiener + perceptual + ODE factor
xWiener + perceptual + vector accumulated solver error
Full-xhat oracle risk
Time-only baseline
```

然后比较：

```text
这些 risk 能不能预测 E5.5 的真实 Y(W)？
```

核心指标：

```text
Spearman correlation
ROC-AUC / PR-AUC
Dangerous window false negative rate
PIS captured at matched refresh budget
time-only controlled correlation
```

如果 xWPCache 风险分数不能显著超过 Raw / SEA / time-only baseline，就不要进入 online E6。

---

### Step D：做 E6-online

只有当 E6-D0 证明 xWPCache risk 能预测 E5.5 的 window PIS，才进入真实 online cache。

Online 版本只用：

```text
当前 x_t
当前 t
当前 step size h
上一次 refresh 的 cached xhat
累计误差状态
```

不使用：

```text
未来结果
E5.5 分段表
人工 call 区间规则
```

---

### Step E：扩大实验和论文验证

最后再跑：

```text
main32 / main64 / main256
matched refresh ratio
matched latency
FID / IS / Precision / Recall
paired LPIPS / DINO / PSNR / SSIM
refresh pattern visualization
failure case analysis
```

## 5. 这版方案和旧版最大的区别

| 问题 | 旧版倾向 | 新版做法 |
|---|---|---|
| E5.5 怎么用？ | 得到经验分段 | 得到 window-level 因果标签 |
| refresh 依据 | call 区间 + max_age | accumulated x-pred solver risk |
| 是否手工指定安全区？ | 有 | 不作为主方法 |
| 是否需要离线诊断？ | 有但不够核心 | E6-D0 是进入 online 前的必要门槛 |
| 是否能说数学动机？ | 较弱 | 从 x-pred clean prediction error、Wiener proxy、velocity conversion 推导 |
| 如何面对 reviewer？ | “为什么这个区间？” | “risk score 与 causal PIS 的相关性、AUC、消融如何？” |

---

## 6. 最重要的原则

后续所有实验都遵循一个原则：

> **E5.5 的结果不能直接写进 online 策略。E5.5 只能用于验证、选择、校准 xWPCache 的数学风险模型。**

如果我们违反这个原则，方法就会退化成经验 schedule。

如果我们遵守这个原则，论文故事就会变成：

```text
Observation:
    连续 cache 复用会产生非线性累积感知损伤。

Theory:
    对 x-pred 模型，cache 误差来自 clean prediction drift，并通过 velocity conversion 进入 ODE 更新。

Method:
    用 Wiener clean proxy + perceptual weighting + accumulated solver error 做 refresh。

Validation:
    用 E5.5 的因果 window PIS 证明 risk score 能预测真实感知损伤。
```

这才是更靠谱、更像论文方法的路线。
