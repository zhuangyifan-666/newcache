# 09｜失败诊断与下一轮修改：如果 xWPCache-v2 不如预期怎么办？

这份文档很重要，因为新方法很可能第一次不会完美。你要学会从失败里判断该改哪里，而不是盲目调参。

---

## 1. 先定位失败发生在哪一层

xWPCache 有三层：

```text
Layer 1: 目标是否正确？
    clean prediction drift 是否能解释 final PIS？

Layer 2: proxy 是否正确？
    Wiener xbar 是否接近当前 xhat 的感知位置？

Layer 3: online 控制是否正确？
    risk 阈值和累计规则是否产生好 schedule？
```

不同失败对应不同修改。

---

## 2. 情况 A：Full-xhat oracle 也预测不好 E5.5

现象：

```text
R_oracle_xhat_vector 和 E5.5 PIS 的相关性低。
```

说明：

```text
最终损伤可能不只由 xhat drift 决定。
```

可能原因：

```text
1. 连续 skip 改变了后续 x_t，非线性误差很强。
2. 感知损伤和中间 xhat drift 不是简单单调关系。
3. sampler corrector 的局部曲率很重要。
```

下一步：

```text
1. 加 curvature risk：比较连续 call 的 xhat 变化方向。
2. 用 E5.5 的 skip trajectory 而不是 full trajectory 计算 risk，对比差异。
3. 考虑 prefix probe，而不是只用 Wiener proxy。
```

---

## 3. 情况 B：Full-xhat oracle 好，Wiener proxy 差

现象：

```text
R_oracle 很好，但 R_xwp 很差。
```

说明：

```text
clean prediction drift 是好目标，但 Wiener proxy 不够好。
```

下一步优先级：

```text
1. 调 beta / f0 / proxy_size。
2. 尝试 G_raw vs G_norm。
3. 尝试直接对 x_t 做 learned-free denoising：Gaussian blur pyramid / wavelet bands。
4. 尝试 prefix-probe：只跑前 K 个 blocks 预测 clean perceptual position。
```

不要马上训练复杂模型。先把 proxy 的无训练版本调清楚。

---

## 4. 情况 C：xW 比 raw 好，但不如 SEAInput

说明：

```text
Wiener clean proxy 有一点帮助，但 SEAInput 的内部 feature 更强。
```

这不是坏事。

你可以做 hybrid：

\[
R = R_{xwp} + \lambda R_{sea}
\]

但论文主线要谨慎：

```text
xWPCache 是 x-pred clean-prediction risk；
SEAInput 是 feature-side redundancy signal；
二者可以互补，但不能让 SEAInput 成为唯一有效成分。
```

消融必须显示：

```text
xW alone 有贡献；
xW + SEA 更好。
```

---

## 5. 情况 D：time-only 和 xW 差不多

这是危险信号。

说明方法可能只是在复现 timestep pattern。

下一步：

```text
1. 做 controlled correlation：控制 start_call/window_len 后 xW 是否仍有预测力。
2. 增加样本数 main32/main64，看内容差异是否出现。
3. 选同一 call 区域内不同样本的 windows，比较 xW 是否能区分失败/安全。
4. 如果不能区分，说明 proxy 不够 content-adaptive。
```

论文里不能只靠 time pattern，否则创新不足。

---

## 6. 情况 E：离线 E6-D0 好，online E6 差

可能原因：

```text
E6-D0 在 full trajectory 上算 risk；
online skip 后 x_t 分布变了。
```

下一步：

```text
1. 保存 online trajectory，重新计算 risk vs final damage。
2. 比较 full x_t 和 online x_t 的 drift。
3. 降低 delta，加入 trust-region。
4. 用 vector residual 而不是 scalar risk。
5. 做 iterative calibration：用 online pilot 的失败 window 更新阈值，但不要手工分段。
```

---

## 7. 情况 F：质量很好，但加速不明显

说明 risk 太保守。

可能原因：

```text
delta 太低
uncertainty 太大
age penalty 太大
proxy distance scale 太大
```

下一步：

```text
1. 提高 delta。
2. 降低 eta/mu。
3. 检查 relative L1 分母。
4. 报 matched quality 下的 speedup，而不是只追 RR=0.3。
```

---

## 8. 情况 G：速度很好，但局部纹理坏

可能原因：

```text
LPIPS-like 中高频权重太弱。
```

下一步：

```text
1. 增大 lambda_L。
2. 调 W_L 的 band-pass 范围。
3. 单独看 LPIPS PIS 的 PR-AUC。
4. 增加 tail 的 uncertainty 或 global trust cap，但不要手工按 call 段。
```

---

## 9. 情况 H：速度很好，但语义结构坏

可能原因：

```text
DINO-like 低频/中低频权重太弱。
```

下一步：

```text
1. 增大 lambda_D。
2. 让 W_D 不只是极低频，而覆盖中低频。
3. 单独看 DINO PIS 的 PR-AUC。
4. 对 false negative 做可视化，确认是不是结构错误。
```

---

## 10. 最终兜底方案：prefix perceptual probe

如果所有无训练 proxy 都无法接近 full-xhat oracle，可以考虑：

```text
只跑前 K 个 transformer blocks，得到 prefix feature。
用一个轻量 head 或 fixed projection 估计 clean perceptual position。
```

这会增加开销，但比完整 forward 便宜。

不过这应该是第二阶段，不要一开始就做。

---

## 11. 一句话总结

> **失败时不要盲目调 threshold。先判断是 clean prediction 目标错、Wiener proxy 错、累计方式错，还是 online 分布漂移。每种失败对应不同修改。**
