# 08｜评测协议和结果表格：怎样证明 xWPCache-v2 真的有效？

这份文档讲最终实验怎么评测。重点是：

```text
不要只报速度。
不要只报一张图。
必须在 matched refresh ratio / matched latency 下比较质量。
```

---

## 1. 实验阶段

### 1.1 Pilot

```text
main8
目的：验证代码和趋势
```

### 1.2 Small main

```text
main32 或 main64
目的：决定是否值得扩大
```

### 1.3 Paper main

```text
main256 / main512 / main1k
目的：论文主表
```

如果算力有限，至少要做到 main64 + 清晰 ablation。

---

## 2. 必须比较的方法

### 2.1 reference

```text
No-cache full compute
```

这是所有 paired metrics 的参考。

---

### 2.2 cheap baselines

```text
Uniform schedule
Fixed stride
Early-refresh fixed schedule
```

注意：early-refresh fixed schedule 只能作为 baseline，不能作为你的主方法。

---

### 2.3 dynamic baselines

```text
RawInput-online
SEAInput-online
TeaCache-style accumulated distance
SeaCache-style SEA filtered distance
```

如果实现完整 SeaCache 很难，至少要实现 SEAInput-online，并清楚说明它是 SeaCache-style baseline。

---

### 2.4 proposed

```text
xWPCache-v2 scalar
xWPCache-v2 vector
xWPCache-v2 vector + uncertainty
```

最后选一个主方法。

---

### 2.5 oracle upper bounds

```text
E5 oracle single PIS schedule
E5.5 window PIS oracle
Full-xhat risk oracle
```

这些不是可部署方法，但可以说明上限。

---

## 3. Matched Refresh Ratio

必须在相同 RR 下比较。

推荐：

```text
RR ≈ 0.50
RR ≈ 0.40
RR ≈ 0.30
```

RR 定义：

```text
refresh_ratio = 实际 full denoiser calls / 原始 full denoiser calls
```

比如 Heun 50 steps 有 99 calls，refresh 50 次：

```text
RR = 50 / 99 ≈ 0.505
```

---

## 4. Matched Latency

RR 不等于真实速度，因为 xWPCache 有 FFT/proxy 开销。

所以还要报：

```text
latency per image
speedup
FLOPs or forward calls
proxy overhead
```

表格：

| method | RR ↓ | latency ↓ | speedup ↑ | proxy overhead | LPIPS ↓ | DINO ↓ | PSNR ↑ | SSIM ↑ |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| full | 1.00 |  | 1.0x | 0 | 0 | 0 | inf | 1 |
| uniform | 0.50 |  |  | 0 |  |  |  |  |
| SEAInput | 0.50 |  |  |  |  |  |  |  |
| xWPCache-v2 | 0.50 |  |  |  |  |  |  |  |

## 5. Paired metrics

对每个 sample，用相同 seed 生成：

```text
full image
cached image
```

然后计算：

```text
LPIPS(full, cached)
DINO distance(full, cached)
PSNR(full, cached)
SSIM(full, cached)
```

这些指标是 paired fidelity，衡量 cache 是否保持原 full trajectory。

---

## 6. Distribution metrics

如果样本数足够，报：

```text
FID
IS
Precision
Recall
```

但 main8/main64 不适合报 FID。FID 至少要几千张才稳。

对于本科生算力有限，论文前期先用 paired metrics，后期再补 FID。

## 10. Refresh pattern 可视化

画：

```text
x-axis: call_index
color: refresh probability across samples
```

要求：

```text
xWPCache 不应该完全固定。
不同样本应该有一定差异。
```

如果所有样本 schedule 几乎一样，说明方法可能退化为 time-only。

---

## 11. 失败样例可视化

保存：

```text
full image
xWPCache image
SEAInput image
uniform image
error heatmap
schedule plot
risk curve
```

每个失败样例都要问：

```text
risk 是否低估？
proxy 是否错？
uncertainty 是否不够？
是否发生长 window 非线性累积？
```

---

## 12. 一句话总结

> **最终实验要证明两件事：离线上，xWPCache risk 能预测 E5.5 因果损伤；在线上，在相同 refresh ratio/latency 下，xWPCache 比 raw/SEA/uniform 更接近 full trajectory。**
