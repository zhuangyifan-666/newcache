# E8｜最终整理：把新思路变成论文证据链

## 1. 新论文故事线

不要再把论文主线写成：

```text
我们用 SEAInput 预测 PMA。
```

这个故事不够稳。

新的主线应该是：

```text
对于 x-prediction diffusion，cache 的核心问题不是特征有没有变化，
而是 skip 某次 denoiser call 会不会造成最终图的感知损伤。

我们先用因果干预实验定义这个感知损伤，
再设计 online 方法去逼近它。
```

这条线更强。

---

## 2. 最终方法可以叫 CPC

建议命名：

```text
CPC = Causal Perceptual Cache
中文：因果感知 Cache
```

其中包含三个版本：

```text
CPC-Oracle：使用 E5 的真实 PIS，不能部署，只作为上限。
CPC-State：直接用 DINO/LPIPS 看当前 x_t，training-free，但可能较慢。
CPC-Prefix：用前缀感知探针预测 PIS 或 DINO(xhat)，作为主方法。
```

论文里可以这样解释：

```text
CPC-Oracle 证明什么叫“正确的 refresh 目标”。
CPC-State 证明直接感知信号比内部 raw feature 更合理。
CPC-Prefix 把直接感知信号变成可部署的加速方法。
```

---


目标 RR：

```text
0.30 / 0.40 / 0.50
```

---

### 表 2：Matched latency 下的质量比较

只 matched RR 还不够。

因为 CPC-State 可能要在线跑 DINO/LPIPS，会增加开销。

所以还要做 matched latency：

```text
在相同秒数 / 相同 speedup 下，谁的 LPIPS 更低？
```

这张表非常重要。

如果 CPC-State 质量好但慢，它在 matched RR 表里可能赢，在 matched latency 表里可能不赢。

这时 CPC-Prefix 就很关键。

---

### 表 3：PIS top-k recall

这个表是新方案的关键证据。

它回答：

```text
不同方法能不能选中真正危险的 call？
```

比较：

```text
Uniform
RawInput
SEAInput
Old PMA
CPC-State
CPC-Prefix
```

指标：

```text
Top-30% PIS recall
Top-40% PIS recall
Top-50% PIS recall
Spearman correlation with PIS
```

如果 CPC-Prefix 在这个表上明显更好，你的论文逻辑就很强。

## 5. 必做消融实验

### 消融 1：没有 noise gate 会怎样？

比较：

```text
CPC-State no gate
CPC-State gate 0.2
CPC-State gate 0.3
CPC-State gate 0.4
```

证明早期噪声阶段不适合直接用感知 metric。

---

### 消融 2：DINO 和 LPIPS 谁更有用？

比较：

```text
DINO-only
LPIPS-only
DINO+LPIPS
```

DINO 偏语义结构。

LPIPS 偏局部纹理。

这和 PixelGen 训练里 LPIPS + P-DINO 互补的思想一致。

---

### 消融 3：prefix depth K

比较：

```text
K=2 / K=4 / K=6 / K=8
```

这回答：

```text
模型前几层到底够不够预测感知风险？
```

---

### 消融 4：probe target

比较：

```text
预测 DINO(xhat_full)
预测 LPIPS/VGG(xhat_full)
预测 PIS_LPIPS
预测 PIS_combined
```

如果预测 PIS 最好，说明“因果感知损伤”比普通 xhat drift 更适合作为 cache target。

## 6. Reviewer 可能问什么？

### 问题 1：你为什么不继续用 SEAInput？

回答：

```text
SEAInput 是一个有用 baseline，但它只是 input-side feature proxy。
我们的 E5 显示 cache 的真正目标是 skip 后的 final perceptual damage。
因此我们用 PIS 定义目标，再用直接感知状态或 prefix probe 去逼近它。
```

---

### 问题 2：PIS 是不是太贵，不能 online？

回答：

```text
PIS 不是 online 方法，而是用于分析和监督的 oracle teacher。
真正 online 的是 CPC-State 和 CPC-Prefix。
```

---

### 问题 3：CPC-Prefix 需要训练，是不是不如 SeaCache training-free？

回答：

```text
我们不重新训练生成模型，只训练一个很小的 probe head。
同时我们保留 CPC-State 作为 training-free 直接感知版本。
CPC-Prefix 的目标是把直接感知判断蒸馏成低开销的 online gate。
```

---

### 问题 4：这个方法是不是只适合 PixelGen？

回答：

```text
它主要面向 x-prediction 模型。
因为 x-prediction 模型天然输出 clean image estimate，适合定义 xhat perceptual trajectory。
这不是缺点，而是本文的专门定位。
```

## 7. 最小投稿路线

如果时间不够，优先级如下：

```text
第一优先级：E5 PIS bank
第二优先级：E6 CPC-State
第三优先级：E7 CPC-Prefix K=4

```

最小可投版本至少要有：

```text

E6：一个直接感知 online cache baseline。
E7：一个更快的 prefix probe cache。
```

先做 E5 的 4 或 8 张图版本。

不要急着优化 online cache。

因为只有 E5 能告诉你：

```text
你到底应该让 online cache 学什么。
```

如果 E5 结果清楚，你后面就不会陷入“凭什么这个 proxy 代表感知变化”的问题。
