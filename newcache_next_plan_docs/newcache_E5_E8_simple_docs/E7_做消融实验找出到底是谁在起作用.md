# E7：做消融实验，找出到底是谁在起作用

> 一句话目标：E6 如果有效，E7 就要证明“为什么有效”；E6 如果不够好，E7 就要帮你找出“哪里坏了”。

---

## 0. 什么是消融实验

消融实验可以理解为“拆零件”。

假设 E6 的 ProxyPMA-online 变好了。reviewer 会问：

```text
到底是 SEA 有用？
还是 timestep 有用？
还是 call_kind 有用？
还是你只是调阈值调得好？
```

E7 的任务就是一个一个拆掉零件，看性能会不会下降。

如果拆掉某个零件后性能明显下降，就说明这个零件真的重要。

---

## 1. E7 要回答的科学问题

E7 主要回答三个问题：

### 问题 1：ProxyPMA 的哪些输入最重要？

比如：

- 只用 SEA 行不行？
- 加 timestep 有没有用？
- 加 predictor/corrector call_kind 有没有用？
- stage-aware 有没有用？

### 问题 2：它是不是只在某个 RR 下有用？

你已经看到 E4 中不同 RR 行为不同：

```text
RR≈0.30：SEAInput-oracle 最稳
RR≈0.40：PMA 和 SEA 接近
RR≈0.50：PMA-no-gate 最强
```

所以 E7 必须分 RR0.30 / 0.40 / 0.50 看，不能只报一个平均。

### 问题 3：它是不是只对少数样本有效？

cache 方法最怕：平均值好，但有几张图崩得很严重。

所以 E7 要看 tail failure，也就是最差样本。

---

## 2. E7 必做的消融组

建议至少做下面 6 组。

### 2.1 Full ProxyPMA-online

这是 E6 的完整方法。

```text
SEA + Raw + timestep + stage + call_kind
```

它是主角。

### 2.2 去掉 Raw

```text
SEA + timestep + stage + call_kind
```

目的：看 RawInput 到底有没有帮助。

你前面 E2/E4 其实已经发现 Raw 不太可靠，所以这个消融可能不会掉太多，甚至可能更好。

### 2.3 去掉 timestep / stage

```text
SEA + Raw + call_kind
```

目的：看“采样阶段”是不是重要。

如果去掉 timestep 后明显变差，说明你的方法不是普通 SEA，而是懂 diffusion trajectory 的。

### 2.4 去掉 call_kind

```text
SEA + Raw + timestep + stage
```

目的：看 Heun predictor/corrector 结构是否重要。

你 E2 已经发现 predictor→corrector 的 distance mass 极大，所以这个消融很关键。

### 2.5 只用 SEA-only proxy

```text
只用 log1p(SEA)
```

目的：确认 ProxyPMA 不是简单换了个名字的 SEAInput。

如果 Full ProxyPMA > SEA-only proxy，说明 E5 学到的附加信息有效。

### 2.6 不使用 accumulated rule

改成每一步单独判断：

```text
risk > threshold 就 refresh
否则 skip
```

目的：证明 SeaCache / TeaCache 类方法常用的 accumulated rule 仍然有用。

大概率 accumulated rule 会更稳，因为有些小变化单独看不大，但连续累积后会变危险。

## 3. E7 可选但很有价值的消融组

如果你有时间，再做这些。

### 3.1 分 stage 训练三个小模型

```text
early 一个模型
middle 一个模型
late 一个模型
```

目的：看不同阶段是不是应该用不同规则。

### 3.2 RR-specific 模型

分别训练：

```text
RR0.30 model
RR0.40 model
RR0.50 model
```

但这要小心，容易被 reviewer 说太复杂。建议作为 appendix，不作为主方法。

### 3.3 cache_age 特征

`cache_age` 表示上一次 refresh 距离现在过了多久。

```text
cache_age = 当前 call index - 上次 refresh call index
```

直觉：很久没刷新了，就算当前风险不大，也可能该刷新。

### 3.4 强制前几步刷新数量

比较：

```text
forced warmup calls = 1 / 3 / 5 / 8
```

SeaCache 论文里很多方法会手动强制前几步刷新，你可以证明你的方法是否依赖这个技巧。

## 4. E7 的输出表应该怎么做

### 4.1 主消融表

建议表格长这样：

```text
Method                    RR    Actual RR   PSNR↑   SSIM↑   LPIPS↓   Worst10 LPIPS↓
SEAInput-online           0.30  ...         ...     ...     ...      ...
ProxyPMA-full             0.30  ...         ...     ...     ...      ...
ProxyPMA-w/o-raw          0.30  ...         ...     ...     ...      ...
ProxyPMA-w/o-stage        0.30  ...         ...     ...     ...      ...
ProxyPMA-w/o-callkind     0.30  ...         ...     ...     ...      ...
ProxyPMA-sea-only         0.30  ...         ...     ...     ...      ...
ProxyPMA-no-accum         0.30  ...         ...     ...     ...      ...
```

同样表格对 RR0.40 和 RR0.50 也做一份，或者合成一个大表。

### 4.2 refresh pattern 表

还要输出不同方法把 refresh 用在了哪里：

```text
Method                    RR    Early share   Middle share   Late share
SEAInput-online           0.40  ...           ...            ...
ProxyPMA-full             0.40  ...           ...            ...
ProxyPMA-w/o-stage        0.40  ...           ...            ...
```

这个表非常重要，因为它能说明你的方法改变了 refresh geometry，而不是只改变了阈值。

### 4.3 predictor/corrector 表

输出：

```text
Method                    RR    Predictor refresh share   Corrector refresh share
SEAInput-online           0.40  ...                       ...
ProxyPMA-full             0.40  ...                       ...
```

因为你前面的 E2 已经发现 Heun call-kind 很关键。

## 5. E7 必须做可视化

### 5.1 RR-quality 曲线

横轴：Actual RR

纵轴可以画：

```text
PSNR
LPIPS
Worst10 LPIPS
```

每个方法一条线：

```text
Uniform
SEAInput-online
ProxyPMA-full
ProxyPMA-sea-only
```

你希望看到的是：

```text
ProxyPMA-full 的曲线在 SEAInput-online 上方，或者 LPIPS 曲线更低。
```

### 5.2 refresh heatmap

画一个热力图：

```text
横轴：call index，从 0 到 98
纵轴：method
颜色：这一 call 被 refresh 的比例
```

这个图能直观看出：

- Uniform 是规则格子。
- SEAInput 是频谱驱动的刷新。
- ProxyPMA 是否更懂 early/middle/late。

### 5.3 worst-case image grid

挑 LPIPS 最差的 8 或 16 张图。

每组显示：

```text
Full reference | SEAInput-online | ProxyPMA-full | 差异图（可选）
```

这个图是给 reviewer 看的：

> 我们不是只在平均指标上赢，也能减少明显坏图。

## 8. E7 怎么判断哪个零件有用

### 8.1 看 Full 是否最好

如果：

```text
ProxyPMA-full > ProxyPMA-sea-only
```

说明你不是简单复制 SEAInput。

### 8.2 看去掉某个零件是否下降

例如：

```text
Full LPIPS = 0.010
w/o callkind LPIPS = 0.016
```

说明 call_kind 有用。

### 8.3 看 RR 分布是否符合 E4 直觉

你希望看到：

```text
RR0.30：ProxyPMA 不要过度浪费 early refresh
RR0.50：ProxyPMA 能利用 perceptual signal，提升 fidelity
```

如果 refresh 全挤在 early，可能说明模型被 early spike 带偏。

---

## 9. 如果 E7 发现 Full 不如某个简化版怎么办

这不是坏事。

例如，如果 `ProxyPMA-w/o-raw` 比 full 更好，说明 RawInput 是噪声，可以删掉 Raw。

最后论文方法应该用最简单、最稳的版本，而不是特征最多的版本。

可以把方法改成：

```text
ProxyPMA = SEA + timestep + call_kind
```

这反而更漂亮：简单、容易解释、容易复现。

## 10. E7 最小可运行版本

如果算力有限，只做下面 4 个方法：

```text
SEAInput-online
ProxyPMA-full
ProxyPMA-sea-only
ProxyPMA-w/o-callkind
```

只跑：

```text
RR0.40
RR0.50
192 samples
```

如果这两档能说明问题，再补 RR0.30 和更多消融。

## 11. E7 写论文时可以形成的贡献句

如果结果符合预期，你可以这样写：

```text
Ablation studies show that timestep and Heun call-kind conditioning are both important for transferring oracle perceptual drift into an online cache policy. Removing either component weakens the latency-fidelity trade-off, demonstrating that our method is not a simple SEAInput rescaling.
```

中文意思：

> 消融实验说明，时间步信息和 Heun predictor/corrector 信息都很重要。去掉它们会变差，所以我们的方法不是简单把 SEAInput 改个名字。

## 12. E7 结束后你应该得到什么

E7 结束后，你应该能回答：

1. 最终在线方法应该用哪些特征。
2. 哪些特征可以删掉。
3. 方法在哪个 RR 最强。
4. 是否减少 worst-case 崩图。
5. refresh pattern 是否符合你的感知流形故事。

得到这些后，就进入 E8：把结果扩大、整理成论文级证据。
